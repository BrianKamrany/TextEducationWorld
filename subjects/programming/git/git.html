1. Overview

Git is a distributed version control system used for tracking changes in the source code of a software project. It has the ability to recover old versions of files. Differentials can show the changes in content between different versions of the same file. The history of all changes made to a file can be viewed. Git makes it easier to share software projects with others. It helps Software Developers work on the same codebase, at the same time, without getting in each other’s way.

There are 2 kinds of commands in Git: Porcelain Commands and Plumbing Commands. Porcelain Commands are intended to be used by the everyday users of Git. Most users will use only the Porcelain Commands. Plumbing Commands are unnecessary for everyday usage. They provide access to low-level functionality and are useful for scripters and graphical user interface developers. Therefore, they will not be covered here.

Git can be used by typing commands on the command line. To achieve this, you can use Command Prompt on Windows, and Terminal on Mac and Linux. Graphical user interfaces are also available for download, which allow you to use Git without having to type the most common commands. These graphical user interfaces implement only a subset of commands, so you may still need to use the command line occasionally. Git comes with a graphical user interface out-of-the-box, which can be accessed by typing git gui on the command line.

Since Git is distributed, it does not require the use of a centralized server for aggregating changes. It is not necessary to have a server at all. However, if a centralized server is desired, Git fully supports being used in that way. There are Porcelain Commands available to help you set up your own server, and you also have the option of using a server on the cloud. Currently, the three most popular cloud providers are GitHub, BitBucket, and GitLab. Whether a server is used or not, you can transfer information in ad-hoc fashion between your Git Repository and another, as long as there is a valid connection between them.

Git is open-source and free for anyone to use. It is available on Windows, Mac, and Linux. It is the most popular version control system used for software projects currently. The second most popular is Subversion.

2. Fundamental Concepts

What is a Git Repository?

Git stores all of the information for a project in a single .git folder, which is located at the root of your project’s directory and hidden by default. This folder is known as the Git Repository. It will contain every version of every file ever added to the Repository, as well as configurations, Branches, and pointers to various Commits. You do not need to manually interact with this folder in most situations. Instead, you should use Porcelain Commands to modify or retrieve the information stored in it. Copying this folder from one place to another would effectively be like creating a full backup copy of the entire Project Repository.

Local File System Operations

In centralized version control systems (not distributed), most commands have to wait for the server to satisfy the request and respond. This means that, at a minimum, commands will have to wait for the network latency between the local computer and the central server to be traversed. On the other hand, when a Git Repository is cloned or initialized, the entire version control history is copied onto the local file system. This includes all of the files and folders that are part of the project. Therefore, most commands in Git operate locally and will perform significantly faster than in centralized version control systems. In addition, work can be completed offline without any issue, since changes do not rely on network connectivity.

Git Workflow

There are three main areas in a Git Project: the Working Tree, the Staging Index, and the Object Database. The Working Tree is all of the files and folders in your project, except for the .git folder itself. In other words, it is what you’re currently working on. The Staging Index is a file that keeps track of which files will be added to the Object Database when you execute the next Commit command. By using the Add command, you selectively choose which files will be added to the Index. The Object Database is where all of the versioned files in the Repository are stored. The Staging Index and The Object Database are located in the .git folder.
New files and folders are added to the Object Database by following a specific workflow. First the files are created or modified. Then they are Added to the Staging Index. After that, all of the files in the Index are Committed to the Object Database at the same time. The Index is then cleared in preparation for the next set of files. This cycle continues perpetually for the lifetime of the project.

Tracking Changes Through Snapshots

Traditionally, version control systems tracked the changes in a file by storing the original file and a list of changes made to it over time. Git does not track changes this way. Instead, it uses the idea of a Working Tree and Snapshots. A Snapshot is a copy of the Working Tree that has been put into the Object Database. Git tracks changes by taking multiple Snapshots of the Working Tree over time, and linking them together in a directed, acyclic graph structure. It takes a new Snapshot only when you command it to do so, using the Commit command. If a file has not changed between Snapshots, then it is shared to avoid duplication. In this way, Git is able to store the entire history of every file in your project.

Tracked Files

Each file in your Working Tree is either Tracked or Untracked. Tracked Files are files that were in the last Commit Snapshot, which means Git knows about them and is tracking them. Files that are currently in the Staging Index are considered temporarily tracked, until they are either Committed to the Object Database or Removed from the Staging Index. Everything else is Untracked. If an Untracked File is deleted or changed, Git will not be able to recover it, since no previous versions of it have been stored.

Hashing

Git relies heavily on Secure Hash Algorithm 1 (SHA-1). A SHA-1 hash is a 40-character hexadecimal string (e.g. 24b9da6552252987aa493b52f8696cd6d3b00373) and is generated based on the content of a file or directory structure. Every Branch, Tag, Commit, and file in Git is dependent on SHA-1. This dependency enables Git to detect file corruption and malicious file changes. It is also a convenient way for you to refer to specific Commits. The only con is that it opens Git up to the very unlikely possibility of a SHA-1 collision. A collision occurs when two different files output the same hash when SHA-1 is applied to them. SHA-1 collisions are effectively impossible due to their statistical improbability. However, if one did happen, then the colliding object would not be saved by Git, because there would already be an object with that hash stored in the Object Database. For more on this topic see the Quote section.

Branches

Most version control systems have the idea of a Branch. Branching is a safe way to diverge from the main line of development. In Git, a Branch is simply a named pointer to a specific Commit. It is most commonly used for experimenting with a new feature. If the experiment is successful, the Branch is Merged back into the main development Branch, incorporating its changes. If the experiment is unsuccessful, the Branch is deleted with no repercussions. You can have as many Branches as you want, and they can point to the same or different Commits. By default, the master Branch is created when a Repository is cloned or initialized, and it functions as the main development Branch.

HEAD

The HEAD pointer is similar to a Branch, but instead of tracking a line of development, it tracks what you currently have in your Working Tree. It does this by pointing to a Branch itself. You decide which Branch it points to by using the Checkout command. When you execute a Checkout, all Tracked Files in your Working Tree are deleted, and the latest Commit Snapshot of the input Branch is copied to your Working Tree. Git will block a Checkout from executing if it detects that you would lose unsaved changes to Tracked Files.
In this way, HEAD is indirectly keeping track of what you currently have in your Working Tree, and it is able to associate your Working Tree back to the Branch that you have Checked Out. When you make a new Commit, HEAD will instruct the Branch it’s pointing at to move its pointer to the Commit you just made. This is how a Branch acquires new Commits and grows.
A Branch is not the only thing that HEAD can point to. If you Checkout a Commit directly by using its hash, or by using a Tag, then you are in a "detached HEAD" state. Commits made in a "detached HEAD" state will eventually disappear, unless a new Branch is made to hold them.

This will work whether the folder contains other files or is empty. The second way is by using the Clone command. This will take a Repository from somewhere else and copy it onto your local file system. The two most popular protocols for Cloning a Repository are HTTPS and SSH.

6. Configuration

After you have created a Repository, you can initialize its settings. Before you can Commit anything, you must set your name and email.

To set your name

git config --global user.name "<NAME>"
git config --global user.name "Brian Kamrany"

To set your email

git config --global user.email "<EMAIL>"
git config --global user.email "BKamrany@example.com"

Your Repository has three different configuration files, of differing scopes. The Local configuration is created per Repository. The Global configuration is created per user on your computer. The System configuration is created for all users on your computer. Local configuration settings take priority over Global configuration settings, which take priority over System configuration settings.

7. Ignoring Files

The .gitignore file is used to ignore certain files in your Repository. This should be used for files that are relevant to your Repository instance, but would not be relevant to clones of your Repository. For example, log files, temporary files, build files, and operating system index files are good candidates to be ignored.

The Ignore File uses regular expression patterns to identify the files and folders to ignore. This makes it easy to ignore an entire directory, or files of a certain type. The Ignore File is usually placed at the root of a Git Project. It should be Committed to the Object Database, just like any other versioned file.

The Exclude File is also used to ignore files. It can be found at the path .git/info/exclude. The difference between these files is that the Ignore File will be available to anyone who clones your Repository, whereas the Exclude File will not be cloned to other Repositories. It is like an Ignore File that applies only to your Repository instance. Therefore, it is neither versioned nor added to the Object Database.

Example Ignore File (Java)

# Compiled class file
*.class

# Log file
*.log

# Package Files
*.jar
*.war
*.ear
*.zip
*.tar.gz
*.rar

8. File Status

Tracked Files in Git exist in one of three states: Modified, Staged, or Committed. Modified means that you have changed a Tracked File, but have not Committed it to the Object Database yet. Staged means that you have marked a modified file, in its current version, to go into your next Commit Snapshot. Committed means that a Snapshot was taken, and the data is now safely stored in the Object Database. If a file has not been integrated into the Object Database or the Staging Index, then it is Untracked and does not have an associated state. The Status command is used to see the state of files in your Working Tree.

To see file status

git status
    On branch master
    nothing to commit, working tree clean
git status
    On branch master
    Changes to be committed:
      (use "git reset HEAD <file>..." to unstage)

            new file:   staged.txt

    Changes not staged for commit:
      (use "git add <file>..." to update what will be committed)
      (use "git checkout -- <file>..." to discard changes in working directory)

            modified:   modified.txt

    Untracked files:
      (use "git add <file>..." to include in what will be committed)

            untracked.txt

To see a more compact version of file status

git status --short
    M modified.txt
    A  staged.txt
    ?? untracked.txt

9. Staging Changes

The Add command is used to mark a modified or new file to be staged for the next Commit. It does this by adding the file’s name to the Staging Index. Add can be used to begin the process of tracking a new file. It can also be used to resolve a Merge Conflict. If you want the changes you made to a file to be stored safely in the Object Database, using this command is the first step.

To add a file to the Staging Index

git add <FILE/FOLDER PATH>
git add docs/staged.txt
git add docs

The Remove command is used to mark a Tracked File to be staged for deletion. It will also delete the file from the Working Tree.

To stage the removal of a Tracked File

git rm <FILE PATH>
git rm DeleteMe.txt
    rm 'DeleteMe.txt'

To stage the removal of a Tracked File, but keep it intact in the Working Tree

git rm --cached <FILE PATH>
git rm --cached UntrackMe.txt
    rm 'UntrackMe.txt'

Once the above command has been Committed, it is effectively the same as untracking a file. Note that --staged, --cached, and --index all refer to the Staging Index.

To stage the renaming or moving of a Tracked File

git mv <FILE PATH> <TARGET FILE PATH>
git mv docs/Readme.txt docs/README.md

10. Committing Changes

The Commit command takes the files listed in the Staging Index and copies them to the Object Database, creating a Snapshot. All of the files listed in the Index are saved at the same time. In addition, a Commit object is created to point to the Snapshot. To refer to the Commit in the future, you can use its SHA-1 hash. This is the second, and last, step needed to copy a file to the Object Database.

To execute successfully, the Commit command requires you to input a message. The message should explain the changes that are being included. If the command succeeds, then the Staging Index will be cleared in preparation for the next set of changes.

To save all Staged Files to the Object Database

git commit -m "<MESSAGE>"
    [master a00962f] <MESSAGE>
     1 file changed, 1 insertion(+)
     create mode 100644 staged.txt

If you made a mistake in your most recent Commit, you can fix it by using an Amend Commit. This enables you to add Unstaged Files to the most recent Commit, in case you forgot them. It can also be used to fix an incorrect Commit Message.

To modify the previous Commit made:

git commit --amend

11. Tags

The Tag command is used to create a Tag. A Tag is a named pointer to a specific Commit. It always points to the same Commit once created. It is commonly used to keep track of release versions.

To list all known Tags

git tag

There are two types of Tags in Git: lightweight and annotated. A Lightweight Tag only points to a specific Commit. It does not contain any other information. An Annotated Tag stores the tagger’s name and email, the current datetime, a message, and a reference to a specific Commit. An Annotated Tag can also be signed with GNU Privacy Guard.

By default, the Tag command will create a Tag that points to the most recent Commit in your current Branch.

To create a Lightweight Tag

git tag <TAG NAME>
git tag v2.7.1-RELEASE

To create an Annotated Tag

git tag -a <TAG NAME> -m <MESSAGE>
git tag -a v1.0.0 -m "JSimpleFileBackup - Initial release."

To create a Tag by specifying a particular Commit

git tag -a <TAG NAME> <COMMIT HASH>
git tag -a "Terrible_Bug" ab8d78fb

To create a new Branch based on a Tag

git checkout -b <BRANCH NAME> <TAG NAME>
git checkout -b dev v2.0.0

12. Differentials

The Diff command is used to see the changes in content between the last Commit Snapshot and the current Working Tree. In software projects, it is used to see the code changes that have been made since the last Commit. It only shows changes in Tracked Files. You can optionally input a file to restrict the differential to that file’s content.

To execute a differential

git diff
    diff --git a/HelloProvider.java b/HelloProvider.java
    index b78add5..ad7a968 100644
    --- a/HelloProvider.java
    +++ b/HelloProvider.java
    @@ -1,3 +1,6 @@
     public class HelloProvider
     {
    +       public static String getGreeting() {
    +               return "Hello World!";
    +       }
     }
    diff --git a/Main.java b/Main.java
    index a77861e..425122a 100644
    --- a/Main.java
    +++ b/Main.java
    @@ -1,6 +1,6 @@
     public class Main
     {
            public static void main(String[] args) {
    -               System.out.println("Hello World!");
    +               System.out.println(HelloProvider.getGreeting());
            }
     }

To execute a differential on a particular file

git diff <FILE NAME>
git diff modified.txt
    diff --git a/modified.txt b/modified.txt
    index cc802cf..d307236 100644
    --- a/modified.txt
    +++ b/modified.txt
    @@ -1 +1 @@
    -<MODIFIED>
    +<MODIFIED CONTENT>

To execute a differential based on the Staging Index instead of the Working Tree

git diff --staged
    diff --git a/staged.txt b/staged.txt
    new file mode 100644
    index 0000000..7613a42
    --- /dev/null
    +++ b/staged.txt
    @@ -0,0 +1 @@
    +<STAGED CONTENT>

You can also use a graphical diff tool

git difftool

13. Removing Changes

At some point, you may realize that you would like to unstage a file from the Index, or undo the changes made to a file. The Reset command can be used to unstage a file. This will remove a file’s name from the Staging Index but will not affect the file itself in any way.

To unstage all files

git reset
    Unstaged changes after reset:
    D       log.txt
    M       mod.txt
    M       Main.java

To unstage a particular file

git reset <FILE PATH>
git reset Main.java
    Unstaged changes after reset:
    M       Main.java

The Checkout command with the -- option can be used to undo the changes made to a Tracked File. This will revert the file in your Working Tree back to the version contained in the most recent Commit Snapshot. Once this command is executed, the changes cannot be recovered.

To undo the changes made to a file

git checkout -- <FILE PATH>
git checkout -- mod.txt

You may want to clear your Working Tree of all files that are untracked. This is a dangerous operation, since Untracked Files are not stored in the Object Database. There will be no way to recover Untracked Files through Git once they’re deleted.

The Clean command can be used to delete Untracked Files from the Working Tree. The -n option is used to perform a dry run. It will output what would have been deleted had you actually executed the command. The -f option is used to actually execute the command.

To perform a dry-run of a Clean

git clean -n
    Would remove Untracked.txt

To delete Untracked Files from the Working Tree

git clean -f
    Removing Untracked.txt

To delete Untracked Files and Folders

git clean -f -d

To delete Untracked and Ignored Files

git clean -f -x

In addition to deleting files, you can also delete Commits. The Reset command with the --hard option will delete entire Commits.

To completely eliminate the last Commit and revert all Modified Files to their previous committed state

git reset --hard HEAD~
    HEAD is now at e821bb2 Initial commit message.

In Git, you can refer to a Commit using a short hash, which is four or more unambiguous characters from the beginning of the SHA-1 hash (e.g. 24b9da). You can also use a Branch or Tag name. In some cases, it’s easier to use the HEAD pointer, and use an ancestor reference on it. The ^ character means the parent, so HEAD^ is one Commit behind the current Commit that HEAD is pointing to. The ~ character refers to 1 behind, and ~5 refers to 5 behind. In the above command, we are telling Git to Reset to one Commit behind the current HEAD. This permanently deletes the last Commit in HEAD.

14. Branching

A Branch is a named pointer to a specific Commit. When you create a new Commit, the Branch you’re on will begin pointing to that Commit. This is how a Branch acquires new Commits and grows. You can create a new Branch and give it a name using the Branch command. When this command is executed, the Branch that is created is based on the Commit that is currently Checked Out.

Branching is used to safely diverge from the main line of development, and it is most commonly used for experimenting with a new feature. The master Branch is the default Branch, used to track the main line of development. It is created when a Repository is initialized and it is Checked Out by default when a Repository is cloned.

To list all the Branches in your Repository

git branch
    * master
    other

This command outputs a * to indicate the Branch that is currently Checked Out.

To create a new Branch

git branch <BRANCH NAME>
git branch other

It is easy to switch from one Branch to another. The Checkout command is used to do this. You can Check Out a specific Branch by using its name.

To switch to a Branch

git checkout <BRANCH NAME>
git checkout other
    Switched to branch 'other'

To create and Checkout a Branch with one command

git checkout -b <BRANCH NAME>
git checkout -b HotkeyRedo
    Switched to a new branch 'HotkeyRedo'

To rename a Branch

git branch -m <BRANCH NAME>
git branch -m new-name

To delete a Branch

git branch -D <BRANCH NAME>
git branch -D other

15. Merging

The Merge command is used to incorporate changes from a particular Branch into the currently Checked Out Branch. A Merge does not destroy either Branch. The same Branches can be merged together multiple times as they continue to progress down their own lines of development.

To Merge a Branch with the currently Checkout Out Branch

git merge <BRANCH NAME>
git merge HotkeyRedo
    Updating 032de10..63773a6
    Fast-forward
     first.txt  | 2 ++
     second.txt | 3 +++
     2 files changed, 5 insertions(+)
     create mode 100644 second.txt
git merge HotkeyRedo
    Auto-merging file.txt
    CONFLICT (content): Merge conflict in file.txt
    Automatic merge failed; fix conflicts and then commit the result.

If both Branches have changes to the same portions of the same file, then Git cannot Merge on its own. This is called a Merge Conflict. When this happens, you can resolve the conflict by editing the file yourself. Git will insert text into the file to help you identify which portions come from which Branch. After you’ve resolved the conflict, Add that file to the Staging Index, and then use the Merge command with the --continue option. If you are not prepared to resolve the conflict, you can cancel the Merge using the --abort option.

Example of a file with a Merge Conflict

text
<<<<<<< HEAD
text
text
text
=======
textChanged
text
textChanged
>>>>>>> other
text

To cancel a Merge due to Merge Conflicts

git merge --abort

To move forward with a Merge after all Merge Conflicts have been resolved

git merge --continue

You can also use a graphical merge tool to resolve conflicts

git mergetool

Once a Merge completes, a new Commit is created to hold the changes that resulted from the Merge. The current Branch then points to that new Commit. In the event that one Branch is a direct successor of another, Git is able to fast-forward the changes. This prevents a new Merge Commit from being created while still incorporating the changes. Sometimes this is undesirable behavior since it hides the history of when a Merge took place.

To Merge and always create a new Merge Commit

git merge <BRANCH NAME> --no-ff
git merge other --no-ff
    Merge made by the 'recursive' strategy.
     first.txt  | 1 +
     second.txt | 1 +
     2 files changed, 2 insertions(+)

16. Rebasing

There is another way to integrate changes from one Branch to another. The Rebase command will also incorporate changes from a particular Branch into the currently Checked Out Branch. It performs this task differently than a Merge does. Instead of creating a Merge Commit to contain the changes between two Branches, it instead takes the input Branch and reapplies all the Commits in your current Branch on top of it. It only applies the Commits that the input Branch is missing. This comes at a cost however. Since a Commit cannot be changed, Git needs to create new Commits for all of the Commits that were missing, so that they can be integrated into the input Branch. Therefore, Git needs to rewrite the history of the project. As long as Rebase is used on Local Branches that have not been shared with others, rewriting history will not cause any problems. This will result in the history appearing to be sequential instead of a union of parallel branches.

To Rebase the current Branch off the input Branch

git rebase <BRANCH NAME>
git rebase master
    First, rewinding head to replay your work on top of it...
    Fast-forwarded master to third.
git rebase master
    First, rewinding head to replay your work on top of it...
    Applying: <MESSAGE>
    error: Failed to merge in the changes.
    Using index info to reconstruct a base tree...
    M       first.txt
    Falling back to patching base and 3-way merge...
    Auto-merging first.txt
    CONFLICT (content): Merge conflict in first.txt
    Patch failed at 0001 <MESSAGE>
    The copy of the patch that failed is found in: .git/rebase-apply/patch

    When you have resolved this problem, run "git rebase --continue".
    If you prefer to skip this patch, run "git rebase --skip" instead.
    To check out the original branch and stop rebasing, run "git rebase --abort".

Just like with a Merge, a Rebase can run into a conflict. If there is one, you can handle it in the same way as a Merge, but you should use the Rebase command instead.

To cancel a Rebase due to conflicts

git rebase --abort

To move forward with a Rebase after all conflicts have been resolved

git rebase --continue
    Applying: other_branch

17. Stashing

The Stash command is used to safely store away your current changes. This can be useful if your Working Tree has uncommitted changes, but you want to switch Branches to work on something else. Using Stash will prevent you from making a Commit before you are ready to do so.

To stash your uncommitted changes

git stash

To stash your uncommitted changes and Untracked Files as well

git stash --include-untracked

To show all stashes:

git stash list

To reapply the most recent stash to the current Working Tree

git stash apply

To reapply a specific stash

git stash apply stash@{<NUMBER>}
git stash apply stash@{2}

To reapply a stash and stage all the previously staged files

git stash apply --index

To delete the most recent stash

git stash drop

To remove a stash from the collection

git stash drop stash@{<NUMBER>}
git stash drop stash@{0}

To Apply and Drop the most recent stash

git stash pop

18. History

The Log command is used to see the history of a Branch. By default, it shows the history of the current Branch in reverse chronological order.

To see the history of Commits

git log
    commit 82d743a79ddf9707afcb9ba88bed6b12604252ec
    Author: abba <abba@doo.com>
    Date:   Sun Oct 22 13:37:38 2017 -0500

        Added new content.

    commit a00962fa3b8f554e1a0877f008cfbb499bec80c7
    Author: abba <abba@doo.com>
    Date:   Sun Oct 22 12:08:23 2017 -0500

        <MESSAGE>

To see the history of Commits with one line per Commit

git log --pretty=oneline
    82d743a79ddf9707afcb9ba88bed6b12604252ec (HEAD -> master) Added new content.
    a00962fa3b8f554e1a0877f008cfbb499bec80c7 <MESSAGE>

To see the last <NUMBER> Commits of the current Branch

git log -<NUMBER>
git log -1
    commit 82d743a79ddf9707afcb9ba88bed6b12604252ec
    Author: abba <abba@doo.com>
    Date:   Sun Oct 22 13:37:38 2017 -0500

        Added new content.

To see the history between two Branches

git log <BRANCH NAME>..<OTHER BRANCH NAME>
git log master..experiment

To see a graphical visualization of the history

gitk

19. Remote Repositories

A Remote Repository is any other Repository that tracks the same project as your Local Repository does. Remotes are usually accessed through the Internet and hosted somewhere else. They can also be hosted on your local computer or a network share. If you cloned your Git Repository using a URL, then a Remote named origin will automatically be configured to point to the originating URL. To be able to collaborate with others on a Git Project, you need to have at least one Remote configured.

To see a list of all Remote Repositories

git remote
    origin
    archive

To add a new Remote

git remote add <REMOTE NAME> <URL>
git remote add pb https://github.com/paulboone/ticgit

To rename a Remote

git remote rename <REMOTE NAME> <TARGET REMOTE NAME>
git remote rename foo cicada3301

To remove a Remote

git remote rm <REMOTE NAME>
git remote rm foo

20. Getting Remote Information

The Fetch command is used to download information about changes from a Remote Repository to your Local Repository. If someone else Pushes a new Branch to Origin (they create a new Branch on a Remote Repository), your Repository would not know about that Branch until you executed a Fetch. This command does not incorporate changes into your Branches. You must do that yourself using a Merge, Rebase, or Pull. This command is only used to gather data.

To gather changes from Origin

git fetch
    Password for 'https://MassiveBrian@bitbucket.org':
    From https://bitbucket.org/MassiveBrian/git_tutorial
     * [new branch]      foobar27   -> origin/foobar27

To gather changes from a Remote

git fetch <REMOTE NAME>
git fetch WorkRepo
    Password for 'https://MassiveBrian@bitbucket.org':
    From https://bitbucket.org/MassiveBrian/git_tutorial
     * [new branch]      lala       -> WorkRepo/lala

The Pull command is used to synchronize a Local Branch with a Remote Branch. All of the changes in the Remote Branch will be incorporated into your Local Branch. The Pull command is equivalent to a Fetch + Merge. You can include the --rebase option to do a Fetch + Rebase instead.

To gather changes from a Remote and Merge those changes into the currently Checked Out Branch

git pull <REMOTE NAME> <BRANCH NAME>
git pull WorkRepo testbr

To gather changes from a Remote and Rebase those changes into the currently Checked Out Branch

git pull --rebase <REMOTE NAME> <BRANCH NAME>
git pull --rebase WorkRepo testbr

To list all Remotes Branches in all Remote Repositories

git branch -r
    WorkRepo/HEAD -> WorkRepo/master
    WorkRepo/foobar27
    WorkRepo/lala
    WorkRepo/master

21. Giving Remote Information

The Push command is used to transmit Commits from the Local Branch you have Checked Out to a Remote Branch. Other Repositories will be oblivious to the Local Commits you have made until you Push them.

To insert your changes into a Remote Branch

git push <REMOTE NAME> <REMOTE BRANCH NAME>
git push origin master

If you try to Push changes that are out-of-date with the Remote, your Push will be rejected. This will prevent you from overwriting a newer Commit.

git push origin master
    Password for 'https://MassiveBrian@bitbucket.org':
    To https://bitbucket.org/MassiveBrian/git_tutorial.git
     ! [rejected]        master -> master (non-fast-forward)
    error: failed to push some refs to 'https://MassiveBrian@bitbucket.org/MassiveBr
    ian/git_tutorial.git'
    hint: Updates were rejected because a pushed branch tip is behind its remote
    hint: counterpart.  Check out this branch and integrate the remote changes
    hint: (e.g.  'git pull ...') before pushing again.
    hint: See the 'Note about fast-forwards' in 'git push --help' for details.

To delete a Remote Branch from a Remote

git push <REMOTE NAME> --delete <BRANCH NAME>
git push origin --delete shared_fix
    Password for 'https://MassiveBrian@bitbucket.org':
    To https://bitbucket.org/MassiveBrian/git_tutorial.git
     - [deleted]         shared_fix

By default, the Push command doesnt transfer Tags to Remotes. You have to explicitly Push the Tags after you have created them.

To share a Tag

git push <REMOTE NAME> <TAG NAME>
git push origin v1.5

To share all Tags

git push <REMOTE NAME> --tags
git push origin --tags

22. Tracking Branches

When you Fetch from a Remote, Git makes internal copies of the Branches it finds in the Remote and stores them inside your Local Repository. These branches are called Remote-Tracking Branches. Remote-Tracking Branches cannot be modified directly, but you can create a Local Branch from them, which can be modified as per normal.

To create a Local Branch from a Remote-Tracking Branch

git checkout -b <LOCAL BRANCH NAME> <REMOTE NAME>/<REMOTE BRANCH NAME>
git checkout -b Ticket-5124 origin/Ticket-5124

Creating a Local Branch from a Remote-Tracking Branch automatically converts that Local Branch into a Tracking Branch, connecting it to the Remote-Tracking Branch. The Remote Branch that it intends to synchronize with is called the Upstream Branch. Tracking Branches are Branches that have an indirect relationship to an Upstream Branch. If youre on a Tracking Branch and Pull without inputting a Remote or Branch, Git automatically knows which server to Fetch from and which Branch to Merge. A Push behaves similarly.

When you clone a Repository, it automatically creates a Master Branch which tracks origin/master by default.

To see all configured Tracking Branches

git branch -vv

To share the changes in the currently Checked Out Branch with the Upstream Branch

git push

To gather changes from an Upstream Branch and Merge those changes into the currently Checked Out Branch

git pull

To gather changes from an Upstream Branch and Rebase those changes into the currently Checked Out Branch

git pull --rebase

23. Cloud Providers

The three most popular centralized Git Server providers are GitHub, BitBucket, and GitLab. If you work in a distributed team or at a company, it may be worthwhile to set up your project on one of these cloud providers. This will provide a common access point to the Project Repository for all team members to synchronize their changes with and against. Repositories on the cloud are usually thought of as the authoritative version of the project.

On these providers, you are given the option to Fork a Repository. When you Fork a Repository, a copy of it will be placed into your own project space for you to do with as you please. This feature is commonly used in open source projects. Forking enables you to immediately start modifying a project without waiting for the owners to give you permission to do so. However, since the forked project is a separate copy, your changes will not affect the original project.

If you would like to affect the original project, you can make a Pull Request. If the owner of the original project accepts your Pull Request, then the changes you made will be integrated into the original project for everyone to enjoy.

24. Subversion

To aid in the transition from Subversion to Git, a number of SVN commands have been created. These commands allow you to use Git locally even if your company or team uses Subversion. You will be able to transmit and receive changes as if you were using Subversion. However, using the SVN commands properly requires you to follow certain rules and best practices. This is because Subversion relies on a linear workflow. The rules are not trivial, so these commands are not covered here.

25. Definitions

Alias

A shortcut for one or more Git Commands. It can be used to avoid memorizing syntax.

Annotated Tag

An object that stores the tagger’s name and email, the current datetime, a message, and a reference to a specific Commit.

Branch

A named pointer to a specific Commit. It is most commonly used for experimenting with a new feature.

Branching

A safe way to diverge from the main line of development.

Commit

An object that has a SHA-1 hash and points to a Snapshot.

CVCS = Centralized Version Control System

A system for tracking changes in files that relies on a central server.

DVCS = Distributed Version Control System

A system for tracking changes in files, capable of functioning in ad-hoc fashion.

Detached HEAD

When HEAD is not pointing to a Branch.

Differential

The changes in the content of a file between the last Commit Snapshot and the current Working Tree.

Exclude File

The .git/info/exclude file used to ignore certain files in your Repository. It is unversioned and cannot be Cloned by others.

GPG

GNU Privacy Guard

GUI

Graphical User Interface

Git

A distributed version control system used for tracking changes in the source code of a software project.

Git Project

A project that uses Git for version control. It is usually a software project.

Git Repository = Project Repository = Repository

The hidden .git folder which contains the Object Database, Staging Index, configurations, Branches, Tags, and Commits. It is located at the root of your Git Project.

HEAD

A pointer that tracks what you currently have in your Working Tree. It usually points to a Branch.

HTTPS

Hyper Text Transfer Protocol Secure

Ignore File

The .gitignore file used to ignore certain files in your Repository. It is versioned and can be Cloned by others.

Master

The default name for a Branch.

Merge Conflict

When Git cannot Merge on its own due to the same parts of a file being changed by both of the Branches to be Merged.

Object Database

Where all of the versioned files in the Repository are stored.

Origin

The default name for a Remote Repository.

Patch

The differences between one Snapshot and another.

Plumbing Command

A command intended to be used by scripters and graphical user interface developers.

Porcelain Command

A command intended to be used by the everyday users of Git.

Remote = Remote Repository

Any other Repository that tracks the same project as your Local Repository does.

Remote-Tracking Branch

An internal Git Branch that synchronizes with a Remote Branch.

SCM

Source Control Management

SHA-1 = Secure Hash Algorithm 1

A hash that is a 40-character hexadecimal string (e.g. 24b9da6552252987aa493b52f8696cd6d3b00373) and is generated based on the content of a file or directory structure.

Snapshot = Commit Snapshot

A copy of the Working Tree that has been put into the Object Database.

SSH

Secure Shell

SVN = Subversion

The second most popular version control system currently.

Staging Index = Index = Stage = Cache

A file that keeps track of which files will be added to the Object Database when you execute the next Commit command.

Tag

A named pointer to a specific Commit. It always points to the same Commit once created.

Tracked File

A file that was in the last Commit Snapshot, which means Git knows about it and is tracking it.

Tracking Branch

A Local Branch that is connected to a Remote-Tracking Branch.

Upstream Branch

The Remote Branch that a Tracking Branch synchronizes with.

VCS = Version Control System

A system for tracking changes in files.

Versioned File

A file stored in the Object Database.

Working Tree = Working Area = Working Copy

All of the files and folders in your project, except for the Git Repository folder itself.
